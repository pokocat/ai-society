### 0.3 最本质的那句话怎么落地

> 「社群关系需要一个记录的关系链，基于微信生态去构建社群体系。」

技术上即三个锚点：
1. **人 = unionid**（微信开放平台绑定后，小程序/公众号/企微客户同一身份；M3 过渡期用 join_way `state` + 手机号兜底，M4 走代开发拿 unionid↔external_userid 映射）——落现有 `member_identifier`（unionid 优先级并档已实现）。
2. **关系 = `referral_relation` 逐边记录**（每人一条 `referrer_id` 边，**深度不限**；lv1/lv2/lv3 物化列仅为 ≤3 级分析加速。生态管理员 ALL 数据范围可见全树）。
3. **群 = 企微客户群**（`community_group.wecom_chat_id`），进退群经官方回调进入现有八态漏斗，聚合数据事务内维护、可对账。

## 1. 合规调研结论（2026-07，来源以官方文档为准）

> 调研方法：微信开放文档 / 企业微信开发者文档 / 帮助中心为「官方书面规则」；第三方 SCRM、律所文章仅佐证并标注。规则时效性强（尤其虚拟支付 2024-2026 剧变），**实施前须复核当日官方页面**。

### 1.1 小程序侧

#### 裂变邀请（需求①的合规边界）

- **允许**：邀请奖励挂钩**被邀请人的真实动作**（注册、完成付费），发放**虚拟权益**（会员时长/成长值/优惠券）。本项目激励=成长值+会员时长，合规。
- **禁止**：强制分享解锁（「分享后才能看/用」）；「分享出去即得利」。处罚阶梯：站内信+3 天整改 → 封分享能力/下架 → 恶意对抗永久封号拉黑主体。
- **高风险不做**：现金返利拉新（诱导分享+拉人头计酬双重触线）。
- 来源：《微信小程序平台运营规范》5.1 滥用分享、官方违规案例集锦（developers.weixin.qq.com/community）。

#### 分销红线（备案，本项目已规避）

- 微信支付仅支持**两级**分销、三级及以上封商户号；《禁止传销条例》三特征（拉人头/入门费/团队计酬）具其一即涉传，刑事立案参考线三级且三十人。
- **本项目决策「只记录不返佣」→ 中台零计酬**，≤3 级关系链纯归因分析，无此风险。若未来引入返佣，只做一级直推、佣金不与下线人数挂钩，且须回到方案层重审。

#### 会员收款：虚拟支付（需求⑤，重大政策变化）

- **2026-04-01 起，小程序内虚拟商品购买全终端（iOS/安卓/鸿蒙/Windows）强制接入「小程序虚拟支付」**，到期未接入判定违规；引导跳外部 H5/网页收款以规避 = 违规下架+封支付能力。
- 现行费率（官方「技术服务费」页，限时优惠中）：**iOS 12%**（全部为苹果佣金，腾讯 5% 暂免；标准 17%）；**安卓/鸿蒙/Windows 主动支付 1%**（标准 10%）；会员订阅首笔 1%/续费 10%。
- 付费社群会员费属可售虚拟品（官方列举含「VIP 会员、录制课程」），类目覆盖文娱/工具/社交/资讯等。
- **「会员订阅」（自动续费）门槛**：接入虚拟支付+开通 iOS 支付+**发布满 90 天+近 30 天日均 DAU≥1 万**+邮件申请；iOS 侧苹果扣费、用户 App Store 退订、开发者无法主动退款。
- **→ M3 决策：先用「主动支付」卖买断周期（月卡/季卡/年卡），会员订阅（自动续费）达标后 M4 再上。**
- 来源：官方虚拟支付文档（platform-capabilities/business-capabilities/virtual-payment，含 iOS 接入页、devplan 费率页、vip 会员订阅页）。

#### 类目与资质（办证关键路径，数月周期）

| 事项 | 要求 | 备注 |
|---|---|---|
| 主体 | **企业主体**（个人主体不能做社交类目/虚拟支付） | 前置 |
| ICP 备案 | 所有小程序强制 | 免费，较快 |
| 类目 | 「社交-社区/论坛」或「教育-知识付费」（按产品核心申报，常组合） | — |
| 《增值电信业务经营许可证》 | 社交-社区类目普遍要求（第二类·信息服务）；含充值/付费会员必须 | **周期数月，立项即启动** |
| 微信开放平台 | 绑定小程序（+公众号）拿 unionid；开发者资质认证一次性 ¥300 | 立项即做 |
| EDI | 仅多商家入驻平台需要 | 本项目不需要 |

#### 归因与身份技术面

- **小程序码**：`getUnlimitedQRCode` scene **≤32 可见字符**（数字/大小写字母/部分符号）；现有邀请码 `FLM-`+12 位共 16 字符**直接放得下**；码数量无上限、永久有效；生成限频 5000 次/分钟 → 为每个会员**预生成**专属码。
- **unionid**：绑定同一开放平台后 `wx.login`+`code2Session` 静默获取（无需授权弹窗）。
- **分享归因边界**：`onShareAppMessage` **拿不到**「分享给了谁/是否成功」——归因只能靠被邀请人打开带参 path/scene 后回传（现有 `ReferralService.bind` 即此模型）；群维度可用 shareTicket→openGId 识别「同一个群」但拿不到群成员。
- **手机号快速验证组件**：¥0.03/次（1000 次免费额度）→ 只在**支付/绑定节点**拉起，不做全员门槛。

#### 触达

- **一次性订阅消息**：用户在关键节点（报名/购买）授权，一次授权发一条，单次最多勾 3 个模板；**不得强制订阅换权益**。开课提醒/入群通知走这条。
- 长期订阅仅政务民生类目，拿不到；服务号模板消息严禁营销群发。

### 1.3 需求④「机器人」的诚实结论

**客户群内不存在任何合规的自动收发消息机器人**（官方底层限制，非资质问题；iPad 协议/Hook 类外挂 = 封号风险 + 违反 CLAUDE.md §17，不做）。「机器人自动分享/讲课/解答」重构为四条合规通道，能力等价：

| 用户想要的 | 合规实现 | 自动化程度 |
|---|---|---|
| 自动分享（新人触达） | 入群欢迎语（素材 API + welcome 事件） | **全自动** |
| 自动分享（日常推送） | 中台排期 → 群发任务 → **群主一键确认** → 回填留痕；**内部群机器人即时提醒群主**（已实现，真实 webhook），确认延迟压缩到秒级 | 半自动（平台强制确认步；官方无绕过 API，2026-07 复核）|
| 讲课 | **企微群直播**（排课→开播→回放 3 年），链接经欢迎语/群发分发 | 全自动排程+真人开播 |
| 解答 | **AI 答疑进自有小程序** + **微信客服 1:1 API 自动应答**（48h 窗口、每轮≤5条，官方全自动，M3c 接入）；M4：会话存档读提问+内部智能机器人辅助员工回群 | 小程序/客服会话内全自动 |
| 运营协同通知 | **内部群机器人 webhook**（官方 91770，无需认证即可用） | **全自动（已实现）** |

| 付费档身份已枚举（pro/premium/black_gold/vip） | 推荐准入（`RecommendEngine:67-97`）不校验付费/有效期；无到期流转作业 |

## 3. 边界变更声明（SPEC §2.2 修订案）

**原边界**：中台不建交易域，订单/佣金/收益只有只读镜像（经 sync 层），提现只做审批协同。

**变更**（用户决策「小程序内收款」触发，按 CLAUDE.md §15「先在方案层面对齐边界」于本文档声明）：

> 交易域**一分为二**：
> 1. **会员费（一方营收）进中台**：新建 `membership_plan`/`membership_order`，小程序虚拟支付收款，中台是该类订单的**事实源**（不再是镜像）。支付回调幂等、状态机管控、审计留痕，与既有护栏（§1 SQL 安全/§5 幂等审计）同规格。
> 2. **项目方交易/佣金/收益维持外置只读**：`order_reference`/`earnings_snapshot` 仍为镜像，提现仍为审批协同。**返佣不进中台**（用户决策「只记录不返佣」）。

随 M3a 实施时同步修订 `CLAUDE.md` §15 措辞（增加「会员费一方订单除外」限定），并在 `ontology/bossclub.ttl` 区分 `:MembershipOrder`（一方）与 `:OrderReference`（镜像）两个概念。

## 4. 数据模型与本体增量（设计稿；实施顺序 ttl → MAPPING.md → 新 Flyway 迁移）

### 4.1 权益域（需求⑤）

**新表 `membership_plan`（会员套餐）**

| 列 | 说明 |
|---|---|
| plan_code / name | 套餐标识与展示名 |
| grant_identity | 授予身份（FK dict `member_identity`：pro/premium/black_gold/vip） |
| duration_days | 权益时长（买断周期） |
| price_cents / ios_price_cents | 定价（iOS 可单独定价对冲苹果税 12%，是否上浮见 §8 开放点） |
| project_scope | NULL=全生态 / 指定项目 |
| status | 上架/下架（dict + CHECK） |

**新表 `membership_order`（一方订单，区别于镜像 `order_reference`）**

| 列 | 说明 |
|---|---|
| order_no（唯一） | 商户单号，虚拟支付 out_trade_no |
| member_id / plan_id / project_id | 归属 |
| channel | ios / android / other（费率与合规路径不同） |
| amount_cents / fee_rate_snapshot | 实付与**当时费率快照**（费率会变，留痕） |
| status | 待支付→已支付→（退款中→已退款）/ 已关闭；`TRANSITIONS` 表管控，禁直改 |
| paid_at / callback_id | 支付回调幂等键（重复回调返回首次结果） |

**权益载体：复用 `member_project_identity`（不新建 entitlement 表）**

- 支付成功回调（事务内）：upsert 身份行 → `identity=plan.grant_identity`、`status=有效`、`valid_until = GREATEST(now, 现 valid_until) + duration_days`（续费叠加）；写会员时间线+审计。
- 每日到期作业：`valid_until < now` → `status=已过期` + 生成运营任务（续费提醒；**不自动踢群**，移群走人工任务，避免误伤）。
- **门控判定（EntitlementService.check）**：`identity ∈ 付费档 ∧ status=有效 ∧ (valid_until IS NULL 视为无效——付费身份必须有有效期)`。

**门控注入点**（三处）：
1. 小程序端点：入群引导/课程/答疑等接口前置校验（无有效权益 → 引导购买页）。
2. `RecommendEngine` 硬性准入 SQL：目标群类型 ∈ 付费档群时要求有效付费身份。
3. `AssignmentService.confirm` 前置校验（防绕过引擎直接 confirm）。

**豁免机制**：`resource_rules` 新列 `paid_gate_enabled`(默认 TRUE)；新 dict 品类 `paid_gate_exempt_group_types`（默认豁免 游客群/体验官群 作转化前置漏斗）。是否严格全门控见 §8 开放点。

### 4.2 代理归属（需求③）

- `community_group` 增列 `owner_member_id BIGINT NULL REFERENCES member(id)` + 索引——「代理名下的群」。
- **推荐引擎新增第 0 级「邀请归属优先」**（在现五级之前）：
  1. 新会员沿 `lv1→lv2→lv3_parent` 找**最近的代理类祖先**（identity ∈ agent/operator/city_partner 且 status=有效）；
  2. 该祖先名下（owner_member_id）**通过硬性准入**的群 → 候选集限定为这些群（组内仍按现有评分排序）；
  3. 无代理祖先或其名下无可承接群 → 回落普通链路（现 `referrer_in_group` 软加分保留）。
- `resource_rules` 增列 `affinity_first BOOLEAN DEFAULT TRUE`（关闭则退化为纯软加分）。
- 人工改选仍走现有 override_reason + 超容量审批，不变。
- 管理端「调整群归属」走审批单（高风险动作，§18 模式）。

### 4.3 企微化改造（需求④载体）

- `community_group` 增列：`wecom_chat_id VARCHAR NULL`（客户群 chat_id，唯一部分索引）、`join_way_id VARCHAR NULL`（活码配置 id）。
- `resource_rules` 增列 `wecom_group_cap INT DEFAULT 200`（平台硬顶；校验 `target_capacity ≤ wecom_group_cap`）。
- **漏斗语义适配（状态机结构不变）**：「待加好友/已加好友」语义变为「待添加企微客户/已添加」（dict 标签与任务文案更新，状态 code 不动，历史数据零迁移）；好友分支判定从个微 `member_wechat_relation` 切到企微客户关系（新表 `member_wecom_relation` 或对现表加 relation_type 判别列——实施时按最小改动定）。
- 个微负载模型（1800/2000/20 群）转 legacy：规则列保留（存量个微期仍在用），推荐引擎对 WX-E 承接链路不再套个微好友上限；`require_personal_cs=FALSE`（既有开关）放宽双客服要求。
- `matchGroupType` 改字典驱动：新 dict 品类 `identity_group_type`（身份→群类型映射），种子迁移现 switch 内容。

### 4.4 运营内容域（需求④）

| 新表 | 用途 | 关键列 |
|---|---|---|
| `welcome_template` | 入群欢迎语模板 | scope（群类型/项目）、content JSONB（text+附件，经 `Json` 序列化）、wecom_material_id（素材库映射）、status |

- **群发执行 = 复用任务域**：`broadcast_plan` 派发 → 按目标群生成 `task_item`（**新 task_type「群发确认」**，dict+CHECK 扩展）→ 群主在企微客户端确认后回填 attempt → 留痕 sent_at。每群每天 1 条的额度约束在派发器内校验（读当日已派计数）。
