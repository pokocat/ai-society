## 1. 系统边界（SPEC §1.2 / §2）

**中台负责**：资源库（账号/员工/群/会员）、项目隔离与资源方案版本、统一会员 ID 与身份合并、分群推荐与任务闭环、负载与容量管控、回访/工单/交接/审批/异常/审计、跨项目统计。

**中台不负责（外部系统职责，SPEC §2.2）**：企微组织与群创建执行、个微客户端操作（人工执行+回填）、会员权益与项目业务主数据、订单支付退款佣金财务、内容渠道投放。

**首期明确不做（SPEC §2.3）**：个微自动化控制、替代企微后台、会计清算、无授权抓取个微数据。

工程含义（v2 相对 v1 的最大修正）：

- 交易数据在中台只有 `order_reference` **只读镜像**（外部订单 ID + 来源系统 + 同步版本），佣金/收益同理只读——中台不产生订单、不算佣金账。
- 加好友/邀请入群是「**任务派发 → 人工执行 → 结果回填 → 事件/人工确认**」模型（SPEC §6.6/§6.7），不是 API 自动化。
- 仓库内提供 `mock-project-system`（模拟外部会员/订单/活动系统）：产生待分配会员、订单事件、收益数据，经标准同步层（§9）进中台——三端演示完整、边界不破，同时把 SPEC §13.2 同步机制真实做出来。

## 2. 总体架构

```
apps/admin-pc(React,设计稿源码为基底)   apps/member-app(Taro)   apps/jinfu-app(Taro)
        │ REST /api/v1（springdoc 导出 openapi.yaml → packages/shared 生成 TS client）
┌───────┴─────────────────────────────────────────────────────┐
│ server：Spring Boot 3.3 / Java 21 模块化单体                  │
│  identity(会员/身份/关系链)   resource(账号/员工/群)           │
│  assignment(推荐/入群/任务)   governance(审批/风险/审计)       │
│  ops(回访/工单/活动/触达)     platform(项目/权限/报表)         │
│  sync(集成/游标/幂等/对账)    gateway(企微/个微适配,首期Mock)   │
├─────────────────────────────────────────────────────────────┤
│ PostgreSQL 16（Flyway 迁移）  ·  进程内缓存 Caffeine           │
└─────────────────────────────────────────────────────────────┘
        ▲ Webhook / API / 定时同步 / 人工导入（SPEC §13.2）
mock-project-system（模拟外部 会员/订单/活动 系统，独立模块）

ontology/bossclub.ttl —— 设计期概念权威（SPEC §4/§5/§13.3 的 OWL 化）+ MAPPING.md（概念→表→DTO 对照）
```

架构决策：

- **单体分域包，不上微服务**：首期单人开发，事务天然跨域（一次分配 = 会员+群+账号+任务同事务），拆分只有成本没有收益；域间只经 service 接口调用，为日后拆分保留边界。
- **首期不引入 Redis/MQ**：预占与并发控制用 PostgreSQL 行锁 + 预占表（§5.3）；异步用 Spring 事件 + 定时任务。降低部署面，验收标准（SPEC §16）没有一条需要中间件。
- **规则外置**：SPEC §7.1 全部默认值（100 人/群、20 群/个微、1800/2000 好友、90% 预警）落 `resource_rules` 表；推荐权重系数同样落库可调。Drools 仅在规则组合复杂时启用，首期用表驱动策略即可（v1 中 Drools 的定位降级，理由：SPEC 规则均为阈值型，表驱动足够、可解释性更强）。

## 3. 技术选型（一行理由）

| 项 | 选型 | 理由 |
|---|---|---|
| 后端 | Java 21 + Spring Boot 3.3 | 会话既定方向；环境自带 JDK21/Maven 3.9 |
| ORM | JPA + 少量原生 SQL（聚合/物化视图） | 关系表多、CRUD 面大；开放问题 #2 可改 MyBatis-Plus |
| DB | PostgreSQL 16 + Flyway | 环境自带；物化路径 + JSONB + 行锁预占全覆盖 |
| 契约 | springdoc-openapi → openapi-typescript | 单一契约源，三端共用 client |
| PC 前端 | 设计仓库源码直接迁入 `apps/admin-pc`（Vite+React+Tailwind+shadcn） | 设计即代码，保真最高、零还原成本 |
| 小程序 | Taro 4 + React，weapp + h5 双构建 | 会话既定；h5 供本环境 Playwright 验证 |
| 鉴权 | JWT + 三层权限拦截器（§7） | SPEC §11.1 要求服务端强制 |
| 本体 | Turtle（Protégé 可开）+ MAPPING.md | 会话既定：设计期权威词典，不上运行时推理 |

## 4. 领域模型与数据库（对齐 SPEC §13.3，8 域 ≈45 表）

> 命名与 SPEC §13.3「建议核心实体」一一对应；★ 为 v2 新增/重构的关键实体。字典统一 `dict_*` 表（身份、群类型、渠道、任务类型…），消除设计稿多组件枚举漂移。

### 4.1 项目域
- `project`：编码/名称/业务类型/负责人/状态/接入方式/服务区域/启停时间/**当前资源方案版本指针**。状态机（SPEC §8.1）：`筹备中→配置中→待发布→运行中→暂停→已结束/已归档`
- `project_integration`：连接方式 `OpenAPI|REST|Webhook|DB只读`、数据权限 `双向|仅读|仅回写`、endpoint、鉴权状态
- ★`project_resource_version`：资源方案版本化，`draft→validated→published`；**发布后不可变，变更开新版本**；高风险变更挂审批单（SPEC §6.3「草稿—校验—发布」）

### 4.2 资源域
- `account`：类型 `手机号|个微|企微|媒体|凭证`、唯一标识、保管人、实际使用人、设备、实名状态、风险状态。状态机（SPEC §8.2）：`库存→待激活→可用→使用中→风险/冻结→待交接→已停用/已归档`；附 `device` 表
- `employee`：工号/部门/岗位/在职状态——**人员与账号解耦**（SPEC §4.3：员工姓名不写死在账号/群上，一律走关系表）
- ★`account_assignment`：账号↔项目多对多授权（含有效期/授权人），替代 v1 的 projectIds 数组
- ★`account_handover`：交接单（原/新保管人、资产清单、关联审批、状态）

### 4.3 群域
- `community_group`：建群企微 FK、群类型、城市/区域、目标容量、当前人数（聚合列，见 §5.4）、项目归属、二维码版本、状态机（SPEC §8.3）：`待建群→待配置→可承接→服务中→容量预警→已满/冻结→待交接→已归档`
- ★`group_staffing`：一群双客服编组 = group × role(`企微客服|个微客服`) × employee × account，支持主责/协同（SPEC §5「群—企微客服/个微客服」）
- ★`group_qrcode`：二维码版本/图片/有效期/轮换记录；★`group_lifecycle_event`：建群/满员/冻结/交接/归档事件流水

### 4.4 会员域（统一身份，会话本体核心）
- `member`：统一会员 ID（U-xxxxx）、姓名、手机、城市、来源
- ★`member_identifier`：可合并身份标识（类型 `手机号|个微号|unionid|企微external_userid|订单用户ID|报名ID` + 值 + 来源系统），合并策略见 §5.5
- ★`member_project_identity`：会员×项目身份（`游客|体验官|PRO|尊享官|代理|城市合伙人|学员`，字典驱动），含有效期/来源/状态/变更历史（SPEC §4.6）
- `referral_relation`：关系链物化路径 `lv1/lv2/lv3_parent`，绑定时服务端校验 ≤3 级/无环/单推荐人；影响力 = 按（层级×身份）聚合视图（对应金服端影响力矩阵；SPEC §10 档案「关系链」项）
- `member_timeline`：档案时间线，回访/沟通/订单/身份/入群事件统一追加（SPEC §10、§16-6）

### 4.5 关系与分配域（v2 核心增量）
- ★`member_wechat_relation`：会员↔个微/企微好友关系（状态、确认方式 `同步|人工回填`、最近确认时间）
- ★`member_group_assignment`：入群全过程记录——会员/项目/目标群/命中规则/推荐分/分配方式 `AI|人工`/覆盖理由/操作人/好友状态/邀请状态/入群状态/失败原因/时间戳组。状态机（SPEC §8.4）：`待匹配→已推荐→待确认→待加好友→已加好友→待邀请→已邀请→已入群`，异常分支：匹配失败/好友申请失败/邀请失败/拒绝入群/重复入群/人工取消/群已满。**修正 SPEC §14.2 缺口：分配不再只改群人数**
- ★`capacity_reservation`：并发预占（对象类型 `群容量|个微好友额度`、占用量、TTL、关联 assignment），SPEC §7.2「负载预占」

### 4.6 任务域（SPEC §4.8 四表）
- `task_batch`（批次来源：分配/交接/整改/二维码轮换）→ `task_item`（类型 `加好友|邀请入群|回访|工单|交接|群码更新|同步异常|配置整改`，负责人/优先级/截止；状态机 SPEC §8.5：`待创建→待领取/已分配→处理中→待复核→已完成` + `失败|超时|已取消|需人工`）→ `task_attempt`（执行记录/失败原因）→ `task_evidence`（完成证据：截图/备注/回填字段）

### 4.7 运营域
- `follow_up`（回访：分类/优先级/回写开关，完成即写 `member_timeline`）、`ticket`（7 类型，SLA 按类型 2/4/12/24h，指派/升级）
- ★`order_reference`：**只读镜像**（外部订单 ID/来源系统/金额/状态/同步版本）——中台不产生订单
- `activity_record`、`member_task`（小程序任务）+ `points_ledger`（积分账）+ `growth_ledger`（成长值账，邀请 +288 默认值可配）
- 触达三表 `push_task / push_template / push_log`、标签 `tag / member_tag`

### 4.8 治理域
- `approval`：SPEC §11.2 清单驱动（资源方案发布/超容量分配/跨项目共用/交接/批量导出/删除退款…），detail JSONB + 流转历史
- ★`risk_event`：风险中心（SPEC §12 八类异常源），等级/负责人/截止/状态/关联对象，可一键转任务或审批
- `audit_log`：前后值/操作者/项目/原因/关联审批/IP/结果；**append-only**（数据库层禁 UPDATE/DELETE + 定期归档校验和），SPEC §11.3
- `sync_job / sync_error`：游标/幂等键/失败队列/重试/对账报告（SPEC §13.2）

## 5. 核心机制设计

### 5.1 入群执行闭环（SPEC §6.4–§6.7 服务编排）

```
外部系统/人工导入 ──sync──▶ member + member_identifier（去重合并）
  ▶ member_project_identity（待分配池）
  ▶ 推荐引擎(§5.2)：Top1 + 备选 + 命中规则 + 风险提示
  ▶ 运营确认 / 人工覆盖（必填原因；超容量/跨项目/异常账号/高风险会员 → 走审批）
  ▶ 创建 member_group_assignment（已推荐→待确认→…）+ capacity_reservation 预占
  ▶ 好友关系分支（SPEC §6.6）：
      已是好友 → 邀请入群任务
      非好友   → 加好友任务 ──成功──▶ 自动派邀请任务
      重复/冲突 → 人工核对队列
      超载     → 阻断并推荐备选资源
  ▶ 执行人回填 task_attempt/evidence，或 gateway 收到入群事件（SPEC §6.7：仅两者可置「已入群」）
  ▶ 同事务回写：群人数聚合、个微负载、member_wechat_relation、member_timeline、释放预占、风险指标
```

### 5.2 推荐引擎（SPEC §7.3 五级优先级，确定性算法，无 AI 依赖）

1. **硬性准入**过滤：项目、身份、群状态 ∈ {可承接, 服务中}、账号状态、黑名单、硬上限（好友 2000/群满）
2. **业务匹配**：城市/区域、群类型↔身份、会员阶段、服务语言
3. **负载均衡**评分：群剩余容量、个微群数（/20）、好友数（1800 起降权）、员工工作量
4. **连续服务**加权：已有好友关系、历史服务人员、已有群关系
5. **运营策略**：渠道、活动批次、推荐人、特殊项目规则

输出：`{推荐群, 推荐分, 命中规则[], 风险提示[], 备选群[]}`；权重系数落库可调（`resource_rules` 扩展）。

### 5.3 并发预占（SPEC §7.2）

分配确认时：`SELECT … FOR UPDATE` 锁群行与个微账号行 → 校验 `当前聚合值 + 活跃预占 + 本次 ≤ 上限` → `INSERT capacity_reservation`。预占带 TTL，超时由定时任务释放；同一会员同项目在途 assignment 建唯一部分索引防重复分配。验收对应 SPEC §16-4「达到上限时可靠阻断」。

### 5.4 负载口径（SPEC §7.2 / §13.1）

群人数、个微群数、好友数一律**从关系明细聚合**（物化视图或聚合列+触发器），禁止人工快照作为唯一依据；好友数保留「校准时间」（人工回填场景）。所有统计数字可回溯到明细或事件。

### 5.5 统一会员合并

`member_identifier` 命中即合并，优先级 `unionid > 手机号 > 企微 external_userid > 个微号`；冲突进人工核对队列；合并动作落 `merge_log` 可回滚（SPEC §16-1 唯一权威数据源）。

### 5.6 幂等与审计

写接口支持 `Idempotency-Key`（任务回填/审批/同步必须带）；审计经 AOP 切面自动记录 diff；「审批通过才执行」的操作采用 审批单→回调执行器 模式，避免先执行后补单。

## 7. 权限三层实现（SPEC §11.1）

拦截器链：`JWT 解析 → ① 角色功能位（role_permission：模块 × 查看|创建|编辑|删除|导出|审批）→ ② 数据范围（@DataScope: SELF|PROJECT|REGION|ALL，注入 SQL 条件）→ ③ 资源授权（账号/群/会员级 ACL，按需）`。前端隐藏按钮仅为体验优化，**服务端逐接口强制校验**（SPEC §11.1 原文）。

## 8. 三端实现

### 8.1 admin-pc（设计稿源码为基底，迁入 `apps/admin-pc`）
- 保留：全部组件、三主题（默认纯白黄绿黑）、全部交互
- 改造：新增 `src/api/`（生成 client + hooks）与 `src/store/`；**逐模块把 mock 数组替换为 API，UI 结构不动**
- 补齐 SPEC §14.2 点名的断点：会员分配走 assignment API（不再直接 +1 群人数）、好友任务带执行明细与状态机、回访完成写档案时间线、新建群/微信/账号/交接/权限按钮全部接通、权限与导航/操作联动
- 新增页面：**风险中心**（SPEC §9 建议的统一入口；复用 shared 组件风格）

### 8.2 member-app（Taro，5 Tab 全做）
首页（会员卡/积分/待办 = member_task）· 社群（我的群 + 入群码 group_qrcode + 申请入群 → assignment 流）· 任务（勾选 → points_ledger）· 收益（earnings 只读镜像 + 提现 → 审批协同）· 我的（档案/订单 = order_reference）

### 8.3 jinfu-app（Taro，5 Tab 全做）
总览（收益汇总 + 今日经营）· 收益（周期报表 + 提现 → 审批协同）· 影响力（referral 矩阵 + 收入构成）· 人群（推广码 + 下级列表筛选 = referral chain 查询）· 我的（收款账户）；顶部项目切换器 → 全接口带 project scope 参数
